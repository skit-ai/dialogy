dialogy.workflow.workflow module¶

A workflow is supposed to run tasks that can be anticipated well in advance.

Here are few demo’s where you can see mock workflow in action.

  • RuleBasedSlotFillerPlugin

  • VotePlugin

  • DucklingPlugin

A workflow is a hollow conduit, think of a vertically hanging pipe without any medium. If you were to drop a block of … anything? through it, it would pass through with a thud on the ground (yes we assumed gravity).

Structure¶

A workflow allows flexibility, that’s why. There is very little structure to it. We have:

  • input

  • output

  • plugins

Apart from these, we expect at the core, an inference function with machine learning models. Which ones? A N Y ones. As long as you have the compute, there is no restriction. Use statistical models or DL or a bunch of conditions, your Workflow won’t judge you.

Advantages¶

The Plugin concept takes care of the sauciness 🍅 of this project. Any functionality can be bundled into a Plugin and they are portable over to foreign workflows. A Plugin proxies inputs through an access function (an argument to every Plugin) and relays output through a mutate function (another argument for every Plugin). These two functions define interactions between many many (sic) Plugins without knowing the inner workings of a Workflow.

Take a look at DucklingPlugin, this plugin handles inputs, manages the default json output into neatly bundled BaseEntity and other similar classes. Another plugin RuleBasedSlotFillerPlugin takes care of slot names and the entity types that should be filled within.

If your classifier predicts an Intent with Slots supporting any of those entities, then slot-filling is not a worry.

The aim of this project is to be largest supplier of plugins for SLU applications.

Warning

The Workflow class is not supposed to be used as it is. Ideally it should have been an abstract class. There are some design considerations which make that a bad choice. We want methods to be overridden to offer flexibility of use.

This is a light but fairly flexible workflow for building a machine learning pipeline.

Requirements - A list of pre-processing functions. - A list of post-processing functions.

Abstract classes put constraints on method signatures which isn’t desired because a couple of methods here could use more arguments, say, load_model() requires path and version and in some other cases path, version and language.

  • debug (bool) – log level shifts to debug if True.
