dialogy.postprocess.text.slot_filler.rule_slot_filler module¶

Module provides access to a rule-based slot filler.

class dialogy.postprocess.text.slot_filler.rule_slot_filler.RuleBasedSlotFillerPlugin(rules: Dict[str, Dict[str, Any]], fill_multiple: bool = False, access: Optional[Callable[[…], Any]] = None, mutate: Optional[Callable[[…], Any]] = None, debug: bool = False)[source]¶

Bases: dialogy.plugin.plugin.Plugin

A utility plugin for slot filling <https://nlpprogress.com/english/intent_detection_slot_filling.html>._ An Intent may have a few slots that need to be filled.

This plugin can assist filling pertaining to certain intent:entity:slot-name association rules.

Schema for rules looks like:

{
    "intent_name": {
        "slot_name":"entity_type"
    }
}

This can be represented in a yaml format like so:

slots:
    faqs:
        action_slot: actions
    report:
        time_slot:
        - time
        - number

Let’s run through a practical example. We will create a workflow and preset the output to have expected intent and entities.

In [1]: import yaml

In [2]: from pprint import pprint

In [3]: from dialogy.workflow import Workflow

In [4]: from dialogy.postprocess.text.slot_filler.rule_slot_filler import RuleBasedSlotFillerPlugin

In [5]: from dialogy.types.intent import Intent

In [6]: from dialogy.types.entity import PeopleEntity, NumericalEntity

# ---------------------------------------------------------------------------------------------------
# Create rules
In [7]: rule_string = """
   ...: faqs:
   ...:     number_slot: number
   ...: count_people:
   ...:     number_slot:
   ...:     - people
   ...:     - number
   ...: """
   ...: 

In [8]: slot_rules = yaml.safe_load(rule_string)

# ---------------------------------------------------------------------------------------------------
# ---------------------------------------------------------------------------------------------------
# Setting up the plugin
In [9]: slot_filler = RuleBasedSlotFillerPlugin(rules=slot_rules, access=lambda w: w.output, debug=True)()

# ---------------------------------------------------------------------------------------------------
# ---------------------------------------------------------------------------------------------------
# Creating synthetic values.
# normally you'd get these by running the workflow with models.
In [10]: intent = Intent(name="count_people", score=1)

In [11]: number_entity = NumericalEntity(type="number", range={"start": 0, "end": 7}, body='2 people', latent=False, values=[{"values":2}], dim="number")

In [12]: people_entity = PeopleEntity(type="people", range={"start": 0, "end": 7}, body='2 people', latent=False, values=[{"values":2}], dim="people")

In [13]: entities = [people_entity, number_entity]

# ---------------------------------------------------------------------------------------------------
# ---------------------------------------------------------------------------------------------------
# setting up the workflow to use slot_filler
In [14]: workflow = Workflow(preprocessors=[], postprocessors=[slot_filler], debug=True)

# ---------------------------------------------------------------------------------------------------
# If you notice the instantiation of RuleBasedSlotFillerPlugin,
# you will notice the `access` method expects a Tuple of two elements.
In [15]: workflow.output = (intent, entities)

In [16]: workflow.run(input_="2 am")

Parameters

  • rules (Rule) – A mapping that defines relationship between an intent, its slots and the entities that fill them.

  • fill_multiple (bool) – More than one item be allowed to fill a single slot.

  • access (Optional[PluginFn]) – Signature for workflow access is access(workflow) -> (Intent, List[BaseEntity])

Module contents¶

To handle a single dialog we need a system that can:

  1. Classify inputs into categories.

  2. Extract tokens of additional significance.

  3. Check for associations between the two.

A slot-filler does the third bit. Once an Intent and a set of Entity are identified, there needs to be an arrangement that tells us if any of these are related to each other.

As per the current design, an Intent can contain a number of Slot. A Slot fills an entity of a pre-defined type.

  • A Slot configured to fill a TimeEntity will not fill a LocationEntity.

  • A Slot will not fill more than one entity even if the type matches, unless fill_multiple is provided.
